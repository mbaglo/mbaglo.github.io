Project Overview

This educational project demonstrates how to design and implement a robust dual-servo control system using an STM32 microcontroller. By utilizing structured state machines, PWM timer channels, user command processing over UART, and error management, this project exemplifies embedded system principles for motor control and interactive systems.

Hardware and Peripherals

  • MCU: STM32 (tested on STM32L4 series)
  • Servos: Two generic hobby servos (PWM-controlled)
  • UART: UART2 for user command input/output
  • Timers: TIM2 for dual PWM output; TIM4 for periodic (100ms) interrupt
  • GPIO: Two output pins for status LEDs

Software Architecture

1. State Machine Framework

The core of the servo control project is a finite state machine (FSM) for each servo, as described in state_machine.h/.c. Each servo maintains its own servo_t struct, tracking position, state, loop counters, direction, and error conditions.

  • Servo states include paused, moving, waiting, looping, error and more.
  • Transitions occur on timer events, UART input (user commands), or recipe commands.
  • Error states are triggered by invalid or nested loop commands.

2. Recipe Execution

A "recipe" is a programmatic instruction set stored as a byte array, interpreted at runtime. Recipes can move the servo to precise positions, pause execution, and implement loops or direction reversals.

  • Recipes are defined using macros like MOV, WAIT, LOOP, END_LOOP, REVERSE, and RECIPE_END.
  • Each servo can be assigned a different recipe at runtime.
  • Looping and error detection (such as nested loops) is fully supported for educational purposes.

Example of a recipe (from recipes.c): 

unsigned char recipe1[] = {
    MOV + 0,
    MOV + 5,
    LOOP + 1,
    MOV + 1,
    MOV + 4,
    END_LOOP,
    RECIPE_END
};

3. UART User Commands

UART2 is configured for asynchronous, interrupt-driven communication. The system displays available commands at startup and accepts two-letter inputs, one for each servo. Commands and effects include:

CommandAction
PPause Recipe Execution
CContinue/Resume
R/LMove Right/Left by one position
BBegin Assigned Recipe
XCancel/Reset Recipe
VReverse Current Direction

Upon receiving valid input, the system parses the buffer and applies appropriate state transitions or direct actions to the relevant servo.

4. Timer Logic and Interrupts

  • TIM2 provides PWM generation for both servos on channels 1 and 2.
  • TIM4 triggers every 100ms, acting as the main scheduler for FSM updates and recipe execution.
  • UART reception and timer flags are handled using interrupt callbacks, ensuring a responsive system.

Implementation Highlights

Initialization and Main Loop


int main(void)
{
  // HAL and peripheral initialization
  HAL_Init();
  SystemClock_Config();
  MX_GPIO_Init();
  MX_USART2_UART_Init();
  MX_TIM2_Init();
  MX_TIM4_Init();

  // PWM Outputs
  HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);
  HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_2);

  // Start timer interrupt (100ms)
  HAL_TIM_Base_Start_IT(&htim4);

  // UART reception interrupt (2 chars at a time)
  HAL_UART_Receive_IT(&huart2, (uint8_t *)rx_buffer, 2);

  // Welcome message and command help
  usart2_transmit("Welcome to the Servo Control System!...");

  // Main scheduler loop
  while (1)
  {
      if (timer_flag) {
          timer_flag = 0;
          if (rx_flag) {
              process_user_commands(rx_buffer);
              HAL_UART_Receive_IT(&huart2, (uint8_t *)rx_buffer, 2);
              rx_flag = 0;
          }
          // State machine handling for both servos
          // LED status updates
      }
  }
}

Key Functions Explained

  • process_user_commands() - Parses UART buffer and triggers events for servo state machines.
  • process_servo_recipe() - Steps through a recipe, executing "move", "wait", "loop", or "reverse" operations.
  • update_status_leds() - Sets onboard LEDs to reflect running, paused, or error status for educational observability.
  • set_servo_position() - Maps a recipe position value (0-5) to a corresponding PWM pulse width for the servo.
  • Error handling, such as invalid commands or nested loop detection, transitions the servo to a safe error state and disables further action until reset.

Experimentation and Extension

  • Try writing your own recipes to produce complex motion patterns, including legal and illegal (for educational error handling) instruction sets.
  • Expand the command set, e.g., for direct position movement or synchronizing both servos.
  • Implement blocking or non-blocking waits and experiment with different servo types or hardware timers.
  • Test the system's response to fast, slow, or invalid UART commands to study debouncing and error recovery.

The resulting codebase can serve as a robust starting point for any more advanced robotics, automation, or hands-on embedded system experimentation involving servos, actuators, or interactive control.
